# Contributing a course

Discovery Quest is built so you can add learning content — a new world, a station, a "Learn
it" lesson, or word lists — **without writing engine code**. This guide is for content
contributors. (Format spec: `docs/specs/2026-06-13-course-format-design.md`.)

## What you can contribute (data) vs. what needs an engine change (code)

**You can PR these as data** (YAML), no engine knowledge needed:
- **curriculum** — worlds and stations (each station picks an existing **board** + difficulty
  **bands** + an optional **lesson**);
- **lessons** ("Learn it") — beats: each beat is one narration line + one **view**;
- **content** — the word lists / sentences / pairs a board uses;
- **narration** — the text of every spoken line (you write text; we generate the audio).

**These need a maintainer/engine PR** (and are reviewed as code):
- a **new board kind** (a new kind of interactive question) or a **new view kind** (a new
  lesson visual). If your idea needs an interaction we don't have yet, open an issue first —
  we'll add it to the engine vocabulary, then your content can use it.

To see what board/view kinds exist today — with a one-line description and the fields each
one takes — look at the **engine capability catalog**, `apps/<app>/engine.capabilities.json`
(generated by `npm run course:capabilities`). Each entry lists its `fields` with an `example`
value, so you can copy the shape directly. For full lessons in context, see the reference
exports `docs/specs/course-format/{english,math}.course.yml` (the live courses in the format).

> Math boards are **engine-generated**: a station just picks a topic + `bands`, you don't
> author question data. English boards consume **authored content** (word banks, sentences) —
> each board in the catalog names the `content` list it draws from and the shape of one item.

## The shape of a course (quick reference)

```yaml
course:
  id: english-phonics
  title: English Quest
  subject: English
  companion: luna
  voice: { id: jessica, profile: teaching-slow }
  worlds:
    - id: phonics
      title: Phonics Cove
      emoji: 🐚
      color: "#22D3EE"
      stations:
        - id: ph-set1
          title: First Sounds
          icon: 🅰️
          board: soundToLetter      # must be an existing engine board kind
          bands: [0]                # difficulty band(s)
          lesson: letter-sounds     # the "Learn it" shown first / replayable
  lessons:
    letter-sounds:
      title: Letter Sounds
      sections:
        - id: s
          label: S
          beats:
            - say: lph-2                                   # a narration line id
              caption: This is the letter S. It says sss…  # MUST equal narration[say]
              view: { kind: phoneme, letter: s, word: sun, emoji: ☀️ }  # kind ∈ engine views
  narration:
    lph-2: This is the letter S. It says sss, like at the start of sun.
  content:                # board-specific authored data (optional)
    phonemes: [ { letter: s, sound: /s/, word: sun }, … ]
```

The golden rule: **`caption` must equal `narration[say]`** (what's shown is exactly what's
spoken). `course:check` enforces it.

## How to add things

- **A lesson** — add its beats under `lessons:` and the lines under `narration:`. Use only
  `view.kind`s that already exist (see a reference export). Point a station at it with `lesson:`.
- **A station** — add it to a world's `stations:` with an existing `board`, `bands`, and an
  optional `lesson`. Stations unlock in order (a station opens once the previous earns a star).
- **A world** — add a block under `worlds:` with `id/title/emoji/color` and its stations. Use
  `soon: true` for stations/worlds that aren't built yet (shown locked).
- **Content** — add the board's data under `content:` (e.g. a `phonemes` table, a vocab list,
  cloze sentences) matching that board's schema.

## Validate before you PR

```bash
npm run course:check -- <your-course.yml> --app packages/english
```
It runs two layers:
- **Structural** (JSON Schema, `apps/<app>/course.schema.json`): the course/world/station/
  lesson/beat shape; `board` and `view.kind` must be real engine capabilities; every view's
  props are the right type and the required ones are present (a typo'd prop like `wrod` is
  rejected). This is also the schema an AI should be given to generate a valid course, and the
  one editors use for live autocomplete — add `# yaml-language-server: $schema=...course.schema.json`
  at the top of your file (or point your JSON `$schema` at it).
- **Semantic**: every `beat.say` exists in `narration` and `caption == narration[say]`; every
  `lesson`/`content` reference resolves; every narration line has a generated clip whose baked
  text still matches (no stale audio).

Green = ready to PR. (Note: validation proves your course is *well-formed*, not that the
teaching is *correct* — maintainers still review pedagogy and age-appropriateness on merge.)

## Audio

You write **text** in `narration`. You do **not** need the text-to-speech key — a maintainer
runs `gen-voice` on merge to render new lines (and commits the audio manifest). If you edit a
line's text, `course:check` will flag the audio as stale so we know to re-voice it.

## Pull request & review

1. Fork, add/edit the YAML, run `course:check` until green.
2. Open a PR describing the content (age range, what it teaches).
3. CI runs `course:check`. Maintainers review for **pedagogy, age-appropriateness, and safety**
   — this is a product for young children, so contribution is open but **merge is gated** by
   human review (no external links, no data collection, age-appropriate language/imagery).
4. On merge we render any new audio and ship it.
